# The Bean Brief v2 — Local Political Intelligence & Farmer Event Engine

v2 extends the existing policy-brief pipeline with an **Entity Registry** (who we watch
and how each publishes), entity-driven collection, geo resolution, and a second
farmer-facing brief render. It is built as a **function of the current product** — the
v1 collect → score → triage → brief → deliver pipeline is unchanged; everything below is
additive and gated so the running app is never broken.

## The Entity Registry (the backbone)

Instead of monitoring "all Iowa politics", we watch a curated set of **entities**
(candidates, officials, party orgs) and their **channels** (website, rss, ical, mobilize,
eventbrite, social handle, newsletter list). Everything downstream hangs off this.

- **`registry.json`** is the hand-editable seed (like `watchlist.json`): the IA federal
  delegation, statewide execs, both state parties, and county-party scaffolds. Edit it and
  re-sync — upserts are idempotent. It auto-seeds to `/data` on first boot and syncs into
  SQLite on server start and on every pipeline run.
- **Machine seeders** fill the bulk: `registry-seed openstates` imports all 150 IA
  legislators; `fec` and `socrata` add federal candidates and IA committees when enabled.
- **Attribution** (`registry.resolveEntity`) maps an inbound item to an entity
  deterministically: plus-address tag → external id → platform org id → social handle →
  sender domain. Low-confidence items fall through to the LLM triage step.

## Two-render brief (Track A)

Off the same relevant-item set, the pipeline can produce two renders:
- **Internal** — the existing candid, strategic ISA brief (unchanged).
- **Farmer-facing** — plain, strictly **nonpartisan**, "what this means for your operation".

Enable with `output.farmerBrief: true` in `watchlist.json` (on in the shipped default; a
legacy Pi watchlist without the key stays off so cost never changes silently). It's a
second Sonnet call on busy days only; empty days stay $0. Set `FARMER_BRIEF_TO` to email it,
else it's saved and readable on the web/RSS like any brief.

## What's live now vs. needs a key/account

- **Live with no new setup:** registry (hand-seed), geo, RSS collection, entity-aware
  scoring/triage, two-render brief (save/web), the `/registry` page, all CLI commands.
- **Add a free key:** `OPENSTATES_API_KEY` (done) unlocks the 150-legislator seed;
  `FEC_API_KEY` unlocks federal candidates.
- **Add an account:** the collector Gmail + **App Password** (`EMAIL_INTAKE_*`) unlocks
  email-intake — the highest-leverage capture channel. See `docs/collector-gmail.md`.
- **Counsel first:** the Socrata/IECDB seeder is hard-gated (see Compliance).

## CLI

```
node src/index.js registry-sync            # re-import registry.json into the DB
node src/index.js registry-seed openstates # import IA legislators (needs OPENSTATES_API_KEY)
node src/index.js registry-refresh         # sync + run every seeder whose key is set (fail-soft)
node src/index.js registry-health          # channels that never fetched / are stale
node scripts/subscribe.mjs targets         # subscribe worksheet (per-list plus-addresses)
node scripts/subscribe.mjs confirm         # auto-click double-opt-in confirmation links
```

## Going live on the Pi

1. Ship the release as usual (bump versions, tag, GHCR build, make package Public, Update in Umbrel).
   The Docker build installs the two new deps (`imapflow`, `mailparser`).
2. `registry.json` auto-seeds to `/data`; the server syncs it on boot.
3. Add `OPENSTATES_API_KEY` (and optionally `FEC_API_KEY`) to `/data/.env`, then
   `node src/index.js registry-refresh` (or click **Sync** on the Registry page + run seeders).
4. For email-intake: create the collector App Password, set `EMAIL_INTAKE_*`, flip
   `email_intake.enabled: true` in `watchlist.json`, restart.
5. Verify feed health on the **Registry** page (stale-channel guard) and correct any
   best-effort URLs in `registry.json`.

## Compliance guards (baked in)

- **Nonpartisan by construction.** The farmer render is strictly nonpartisan; the registry
  tracks all parties evenhandedly.
- **Checkoff & PAC — confirmed clear (2026-07-05).** This work is **not** checkoff-funded;
  it's booked on the membership/policy side. ISA runs **no PAC** and does **not** raise money
  for political contributions — so express-advocacy, campaign-coordination, and PAC concerns
  do not apply. The system informs members; it never endorses or solicits.
- **IECDB use-restriction (Iowa Code § 68B.32A(7)).** This governs the *data* (no commercial
  use / solicitation), independent of funding. Because ISA uses the roster only to **inform**
  (nonpartisan briefings) and does not solicit, informational use sits in the statute's
  public-information carve-out. The Socrata seeder still ships behind an explicit
  `IECDB_INFORMATIONAL_USE=true` flag as a light safety default (and needs the dataset id
  anyway) — flip it on after a quick counsel nod on any member-facing use.
- **Member outreach (Track B, future).** Farmer targeting will pull from HubSpot and must
  stay behind a human-approval gate, as informational event packets — not solicitation. Not built yet.

## Not yet built (Track B and beyond)

Event mining (iCal/Mobilize/Eventbrite adapters), cross-source event dedup, farmer event
packets, and HubSpot-targeted outreach. The registry, geo, entity resolution, and `event`
item type are the foundation those will build on.
